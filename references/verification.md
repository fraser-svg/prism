# The Verification Loop

The verification loop runs during CREATING, after every build chunk. It is the
second leg of the triad (translation + **verification** + guidance). Its job:
make sure what was built matches what was intended, and catch fabrications before
the founder discovers them.

## Core Principle: Silent by Default, Interrupt When It Matters

Prism verifies every chunk. The founder only hears about problems. Green chunks
auto-proceed with a brief status message. The goal is ~1-2 interruptions per
build, not 5-10.

## The Precedence Hierarchy

```
Tests PASS + LLM OK      -> auto-proceed, log success, brief status to founder
Tests PASS + LLM flags   -> surface advisory to founder (non-blocking)
Tests FAIL               -> always block, regardless of LLM opinion
User override             -> log override + reason, proceed
```

Tests are ground truth (pass/fail, no hallucination risk).
LLM comparison is advisory (surfaces concerns, does not block on its own).
User judgment is the final override (can dismiss either, but overrides are logged).

## Chunk Verification Flow

After each build chunk completes:

### Step 1: Run acceptance criteria comparison (silent)

Read the acceptance criteria for the current feature from
`.prism/acceptance-criteria.md` (user-facing) and `.prism/test-criteria.json`
(machine-facing). Compare the build output against the machine-layer assertions
using an LLM self-evaluation:

> "Given these acceptance criteria and this build output, does the output satisfy
> the criteria? List any mismatches."

### Step 2: Run tests from machine-layer criteria (silent)

If tests exist for this feature (generated by tdd-guide from test-criteria.json),
run them inline. Do NOT re-invoke tdd-guide unless the fix changes feature scope.

### Step 3: Evaluate and route

| Tests | LLM comparison | Action |
|-------|---------------|--------|
| Pass | No issues | Auto-proceed. Log success. Tell founder: "The {feature} is working. Moving on to {next}." |
| Pass | Flags mismatch | Surface as advisory: "This is working, but I noticed {concern}. Want me to adjust, or is it fine?" Non-blocking. |
| Fail (1st attempt) | Any | Silent fix. Do not interrupt the founder. Attempt to fix the failing test. |
| Fail (2nd attempt) | Any | Interrupt. Surface to founder in plain language: "I've been trying to get {feature} right but something's not landing. Here's what's happening: {plain-language explanation}. Want me to keep trying, or should we rethink this part?" |
| Any | Significant drift | Interrupt as judgment checkpoint: "{Feature} works, but it's doing something different from what you described. {Plain-language diff}. Is this what you meant, or should I adjust?" |

### Step 4: Log the verification outcome

```bash
jq -n --arg action "verification" --arg ts "$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
  --arg feature "{feature}" --arg tests "{pass|fail}" --arg llm "{ok|flagged}" \
  --arg outcome "{auto_proceed|advisory|silent_fix|escalated|override}" \
  '{action: $action, ts: $ts, feature: $feature, tests: $tests, llm: $llm, outcome: $outcome}' \
  >> .prism/history.jsonl
```

## Socratic Rejection UX

When the founder rejects a chunk, their feedback is often vague: "it feels off,"
"that's not right," "something's wrong." Do not ask them to be more specific in
engineering terms. Translate the vibe into engineering changes through follow-up
questions.

**When rejection is vague:**

Via AskUserQuestion:
> "I hear you. Help me zero in on it:"

Options:
- "It's doing the wrong thing" — the feature itself is wrong
- "It's doing the right thing the wrong way" — the approach/UX is off
- "It doesn't feel like what I described" — intent drift
- "I'll try to explain" — free text

Then translate their answer into engineering changes silently. Log the rejection
and translation to history.jsonl.

**When rejection is specific:**

If the founder says something concrete ("the button should be on the left," "it
should send an email, not a notification"), act on it immediately. No follow-up
needed.

**After rejection:**

1. Rebuild the chunk with the correction
2. Re-run the verification loop
3. If it passes, auto-proceed with: "Fixed. {Brief description of what changed}."
4. If it fails again, escalate per the flow above

## Override Logging

Every user override is logged with the reason. This is not punitive. It is
calibration data for tuning the verification sensitivity later.

```bash
jq -n --arg action "override" --arg ts "$(date -u +%Y-%m-%dT%H:%M:%SZ)" \
  --arg feature "{feature}" --arg type "{test_override|advisory_dismissed}" \
  --arg reason "{founder's reason}" \
  '{action: $action, ts: $ts, feature: $feature, type: $type, reason: $reason}' \
  >> .prism/history.jsonl
```

## Test Generation Strategy

Tests are generated ONCE per feature during acceptance criteria generation
(Phase 4). The tdd-guide agent receives machine-layer assertions from
`.prism/test-criteria.json`, not user-layer criteria (vibes).

During verification, run existing tests inline. Only re-invoke tdd-guide if:
- A fix changes the feature's scope (new assertions needed)
- The founder adds requirements during a rejection

This prevents test regeneration overhead on every chunk.

## What the Founder Sees

- Green chunks: "The signup flow is working. Moving on to the dashboard."
- Advisories: "This is working, but I noticed the load time is slower than expected. Want me to optimise, or is it fine for now?"
- Escalations: "I've been working on the payment flow and something's not clicking. Here's what's happening: {explanation}. Want me to keep at it, or should we rethink this bit?"
- Rejections acknowledged: "Got it. Rebuilding that part now."

## What the Founder Never Sees

- Test output, stack traces, or error logs
- LLM self-evaluation reasoning
- "Running tests..." or "Verifying..."
- The word "assertion," "criteria," or "test-criteria.json"
- Any mention of the verification loop itself
