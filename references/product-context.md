# Product Context — 4-File Memory Model

Prism maintains product memory across four files with a clear authority hierarchy.
Three are authoritative (hand-maintained or append-only); one is generated.

## The 4-File Model

| File | Authority | Updated | Purpose |
|------|-----------|---------|---------|
| `PRODUCT.md` | Authoritative, manual | Stage 1 (create), Stage 5 (phases) | Product identity: vision, user, finished product, phases |
| `ARCHITECTURE.md` | Authoritative, semi-auto | Stage 2 (decisions), Stage 3 (drift), Stage 5 (evolution) | Technical shape: stack, patterns, data model, boundaries |
| `DECISIONS.md` | Authoritative, append-only | Stage 2, Stage 3, Stage 5 | Architecture Decision Records. Never edited, only appended. |
| `STATE.md` | Generated, never hand-edited | Every invocation (via `prism-state.sh`) | Current position: active change, stage, blockers, next steps |

**Authority rule:** When PRODUCT.md conflicts with archived specs (`openspec/specs/`),
archived specs are ground truth for what was built. PRODUCT.md is ground truth for
direction (vision, what is next).

**Constraint:** One set of product files per repo. One product per repo.

## When to Create Product Memory Files

### New products (no code, no specs)

Stage 1 Part 0 creates all three authoritative files from templates:

1. Ask product-level sharpening questions (see below)
2. Create `PRODUCT.md` from `templates/product-memory/PRODUCT.md.tmpl`
3. Create `ARCHITECTURE.md` from `templates/product-memory/ARCHITECTURE.md.tmpl`
4. Create `DECISIONS.md` from `templates/product-memory/DECISIONS.md.tmpl`

STATE.md is never created manually. It is generated by `prism-state.sh`.

### Existing code, no product files (Bootstrap)

When a repo has existing code or archived specs in `openspec/specs/` but no PRODUCT.md:

1. Read all archived specs in `openspec/specs/`
2. Inspect the codebase: `package.json`, directory structure, README, key config files
3. Synthesize drafts of PRODUCT.md, ARCHITECTURE.md, and DECISIONS.md from what exists
4. Present draft to user: "I've put together a product overview based on what's already
   built. Take a look and let me know if anything needs adjusting."
5. User confirms or refines, then continue with normal flow

Do NOT ask fresh product-level questions when bootstrap material exists.

### Partial product files

If PRODUCT.md exists but ARCHITECTURE.md or DECISIONS.md do not, create the missing
files from templates and populate from existing PRODUCT.md content (migrate Architecture
section to ARCHITECTURE.md, Architecture Decisions table to DECISIONS.md).

## Product-Level Sharpening Questions

Ask these ONLY when creating a new PRODUCT.md from scratch (no existing code/specs):

1. "What's the full vision for this product? What problem does it solve?"
2. "What does the finished product look like? Describe what a user would see and do."
3. "What's the foundation we need to build first?"
4. "What are the major pieces after that? Help me understand the building blocks."
5. "Does this product have a visual interface?" (determines UI product detection)

If the product has a UI, also ask:
6. "What should it look like? Any design references, styles, or examples you like?"

## When to Read Product Memory

Read silently at every stage transition:
- **Stage 0 (Resume):** Run `prism-state.sh` to generate STATE.md. Read STATE.md for
  fast context recovery. Also read PRODUCT.md for product context.
- **Stage 1 (Understand):** Read PRODUCT.md and ARCHITECTURE.md to inform spec generation.
- **Stage 2 (Plan):** Pass ARCHITECTURE.md to /plan-eng-review.
- **Stage 3 (Build):** Consult ARCHITECTURE.md and DECISIONS.md for build decisions.

Reading is always silent. Never announce "I'm reading your product doc." Just use the
context naturally in conversation.

## When to Update Product Memory

### PRODUCT.md — Updated at product milestones

- **Stage 1 Part 0:** Created from template (new products) or bootstrap (existing code)
- **Stage 5 (Ship):** Update Phases table (status to "shipped"), update "What's Been
  Built" table, suggest next phase

### ARCHITECTURE.md — Updated when architecture changes

- **Stage 2 (Plan):** If /plan-eng-review recommends architecture changes, update sections
- **Stage 3 (Build):** After completion, if architecture changed during build
- **Stage 5 (Ship):** Update to reflect final shipped architecture

### DECISIONS.md — Append-only at decision points

- **Stage 2 (Plan):** Append decisions from architecture review
- **Stage 3 (Build):** Append decisions made during build (e.g., library choices)
- **Stage 5 (Ship):** Append shipping decisions (e.g., deployment strategy)

Each entry follows the ADR format:
```markdown
### ADR-{N}: {Title}
**Date:** {YYYY-MM-DD}
**Context:** {What prompted this decision}
**Decision:** {What we chose}
**Alternatives:** {What else we considered and why not}
**Consequences:** {What this enables and constrains}
**Revisit if:** {Trigger conditions for reconsidering}
```

### STATE.md — Generated, never hand-edited

Generated by `prism-state.sh` on every invocation. Reads from:
- `.prism/registry.json` (stage, workers, checkpoint)
- `PRODUCT.md` (current phase info)
- `DECISIONS.md` (recent decisions)
- `openspec/changes/*/session-context.md` (last session context)

Call: `bash "$SKILL_DIR/scripts/prism-state.sh" "$PROJECT_ROOT"`

If a subagent fails to update any file: retry once, then skip. Never block the user
on a product memory update failure. Log the skip.

## Stale Architecture Detection

At each Stage 1 invocation for product-level requests (not small changes/bugfixes),
the subagent should check for drift by comparing ARCHITECTURE.md against the codebase:

1. **Dependencies:** Compare ARCHITECTURE.md Stack section against `package.json`
   dependencies (or equivalent). Flag new major dependencies not mentioned.
2. **Directory structure:** Check if new top-level directories exist that aren't
   reflected in the Structure section.
3. **Config files:** Check for new config files (`.env`, `docker-compose.yml`, database
   configs) that suggest infrastructure changes.

Do NOT attempt to compare prose descriptions against code semantics. Only flag concrete,
observable mismatches (new package, new directory, new config file).

When drift is detected, ask the user:
"Your architecture doc says {X} but I notice {Y} in the codebase. Want me to update it?"

## UI Product Detection

A product is a "UI product" if ANY of:
- PRODUCT.md Vision section mentions "interface", "UI", "page", "screen", "dashboard",
  "frontend", or "app"
- The codebase has a frontend framework (React, Vue, Next.js, Svelte, etc.)
- The user answers "yes" to "Does this product have a visual interface?"

When a UI product is detected:
- Before first Build: auto-invoke `/design-consultation` if no DESIGN.md exists
- After QA (Verify) stage: auto-invoke `/design-review`
- Product-level questions include "What should it look like?"

## Reading Product Memory in Subagent Prompts

When reading product memory for context, use this shared instruction in subagent prompts:

> "Read PRODUCT.md, ARCHITECTURE.md, and DECISIONS.md in the project root if they exist.
> Extract:
> - Vision and user (from PRODUCT.md)
> - Tech stack and patterns (from ARCHITECTURE.md)
> - Recent decisions (from DECISIONS.md)
> - What's been built and what's next (from PRODUCT.md)
> Use this context to inform your work. Do not contradict architecture decisions
> unless the user explicitly asks to change direction."
