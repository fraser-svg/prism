## Vision

### 10x Check
Prism doesn't just report on itself — it learns from itself. Every session produces a
structured Session Report Card that feeds back into behavior. Bad patterns get auto-detected,
logged as dogfood entries, and used to adjust gate thresholds, review expectations, and
prompt behavior. The self-healing isn't a static report — it's a closed-loop learning system
that makes every Prism installation smarter over time.

### Platonic Ideal
Every session ends with a report card. Reports aggregate into trends. Trends trigger
behavioral changes. The operator sees improvement. The system literally prescribes its
own medicine — and earns trust by being transparently self-critical.

## Accepted Scope: Self-Healing System

### 1. Session Report Card
**Location:** `.prism/dogfood/reports/{YYYY-MM-DDTHH-MM-SS}.json`

**Trigger:** Generated when the gate evaluator processes a transition to `release` (the natural session completion point), OR when the user explicitly runs the bridge command `$BRIDGE session-end`. If the previous session ended in a crash (no report card exists but EventLog has a `session:start` without a matching `session:end`), the resume engine generates a crash-recovery report card with available data before proceeding.

**Schema:**
```json
{
  "schemaVersion": 1,
  "sessionId": "uuid-v4",
  "projectId": "entity-id",
  "timestamp": "ISO-8601",
  "dimensions": {
    "guided_start": { "score": 0-10, "source": "auto", "evidence": "string" },
    "research_proof": { "score": 0-10, "source": "auto", "evidence": "string" },
    "stress_verification": { "score": 0-10, "source": "auto", "evidence": "string" },
    "evidence_quality": { "score": 0-10, "source": "auto", "evidence": "string" },
    "post_handoff_bugs": { "score": 0-10, "source": "manual_on_resume", "evidence": "string | null" },
    "user_corrections": { "score": 0-10, "source": "manual_on_resume", "evidence": "string | null" }
  },
  "overallScore": 0-100,
  "summary": "human-readable one-paragraph assessment",
  "replaySessionId": "links to session replay",
  "crashRecovery": false
}
```

**Scoring rules:**
| Dimension | What it measures | Score 10 | Score 5 | Score 0 |
|-----------|-----------------|---------|---------|---------|
| guided_start | Was the user oriented before planning? | IntakeBrief exists with 0 unresolved questions before plan stage | IntakeBrief exists but has unresolved questions | No IntakeBrief created at all |
| research_proof | Was the approach researched and justified? | SolutionThesis exists with >= 2 alternatives + explicit recommendation reason | SolutionThesis exists with < 2 alternatives OR missing recommendation reason | No SolutionThesis at all |
| stress_verification | Were failure modes tested? | >= 1 stress + >= 1 edge scenario passed in VerificationResult | Only happy_path scenarios ran | No verification scenarios at all |
| evidence_quality | Were claims backed by evidence? | All review findings have evidenceConfidence === "high" | Some findings have "medium" confidence | Any unsupported absence claims (confidence "not_confirmed" with direction "absent") |
| post_handoff_bugs | Bugs found after handoff (manual) | 0 bugs found after handoff | 1-2 minor bugs | 3+ bugs or any critical bug |
| user_corrections | Times user corrected Prism (manual) | 0 corrections needed | 1-2 corrections | 3+ corrections |

**Dimension overlap resolution (reviewer issue #6):** `research_proof` measures whether research was done and justified (SolutionThesis existence + recommendation quality). `evidence_quality` measures whether claims in reviews/verification are properly backed by evidence (from Plan 1's confidence types). These measure different things: research_proof = "did you look before you leapt?", evidence_quality = "are your claims honest?"

**Auto vs manual split:** Dimensions 1-4 are auto-populated from artifact/EventLog queries at session close. Dimensions 5-6 default to `null` until the operator provides input. On next resume, the resume engine prompts: "How did the last session go?" with options: A) No issues, B) Minor bugs found, C) Had to correct Prism. This prompt is skippable — if skipped, dimensions remain `null` and are excluded from the overall score.

**Overall score formula:** `overallScore = (sum of non-null scores / count of non-null dimensions) * 10`. Example: 4 auto dimensions averaging 7.5 = overall 75.

**Crash recovery:** When the resume engine detects an orphan session (session:start without session:end in EventLog), it generates a report card from whatever EventLog events exist. Auto dimensions that can't be scored (insufficient events) are set to `null` with `source: "crash_recovery"`. The `crashRecovery` flag is set to `true`.

### 2. Learning Journal
**Location:** `.prism/dogfood/learning-journal.json`

**Updated:** After each report card is generated. Reads all report cards in `.prism/dogfood/reports/`, aggregates.

**Schema:**
```json
{
  "schemaVersion": 1,
  "lastUpdated": "ISO-8601",
  "totalSessions": 5,
  "patterns": [
    {
      "dimension": "research_proof",
      "trend": "improving" | "stable" | "degrading",
      "avgScore": 3.2,
      "occurrences": 4,
      "recurring": true,
      "firstRecurringAt": "ISO-8601",
      "detail": "Research skipped in 4/5 sessions"
    }
  ],
  "overallTrend": "improving" | "stable" | "degrading",
  "overallAvgScore": 62
}
```

**Recurring threshold:** A pattern is "recurring" after 3+ sessions with score <= 5 for that dimension. Window: all sessions (cumulative). When a pattern first crosses the recurring threshold, `firstRecurringAt` is set and never overwritten — this timestamp is used by HEALTH.md to show "recurring since {date}".

**New-recurring detection:** The journal updater compares each pattern's `recurring` field before and after update. If `recurring` transitions from `false` to `true`, it's "newly recurring" — this triggers auto-dogfood entry generation.

**HEALTH.md alignment:** HEALTH.md shows the last 5 sessions for display, but the Learning Journal uses ALL sessions for pattern detection. These are intentionally different windows — display is recent, learning is cumulative.

### 3. Advisory Prescriptions
**Location:** `.prism/dogfood/prescriptions/{dimension}-{timestamp}.json`

**Trigger:** Generated when Learning Journal detects a recurring pattern (dimension score <= 5 in 3+ sessions).

**Schema:**
```json
{
  "schemaVersion": 1,
  "id": "uuid-v4",
  "dimension": "research_proof",
  "prescription": "Require SolutionThesis with >= 2 alternatives before advancing to spec stage",
  "severity": "low" | "medium" | "high" | "critical",
  "createdAt": "ISO-8601",
  "status": "active" | "resolved",
  "resolvedAt": null,
  "basedOnSessions": 5,
  "patternDetail": "Research skipped in 4/5 sessions"
}
```

**Severity mapping (shared with auto-dogfood):**
| Occurrences with score <= 5 | Severity |
|---|---|
| 3-4 sessions | medium |
| 5-7 sessions | high |
| 8+ sessions | critical |

**Lifecycle:**
- Created: when pattern detected as recurring
- Shown on resume: resume engine reads active prescriptions and appends to `nextRecommendedActions` in Checkpoint: "Advisory: {prescription} (based on {basedOnSessions} sessions, since {createdAt})"
- Resolved: when 3 consecutive sessions score >= 7 on that dimension. Sets `resolvedAt` and `status: "resolved"`.
- Max shown: 3 active prescriptions at resume (highest severity first)
- No dismiss UX in v1 — prescriptions resolve themselves through improved behavior. Dismiss deferred.
- No auto-archiving in v1 — deferred to TODOS.md.

**Behavioral adjustment mechanism:** On resume, active prescriptions are read and injected as human-readable advisory text into the Checkpoint's `nextRecommendedActions` array. This does not modify gates, prompts, or thresholds — it surfaces the prescription as a recommendation. The operator decides whether to follow it.

### 5. Session Replay
**Location:** Extended EventLog events with rich metadata. Timeline queryable via `EventLog.sessionTimeline()` (already exists).

**Session ID:** UUID v4, generated at session start. Stored on the Checkpoint as `currentSessionId` (new field). Cleared after report card generation at session close. The report card generator reads `currentSessionId` from the active Checkpoint to link the report.

**What "rich decision context" means:** Each EventLog entry gains optional `decisionContext` in metadata:
```json
{
  "eventType": "session:gate",
  "summary": "Gate verify->release evaluated",
  "metadata": {
    "sessionId": "uuid-v4",
    "decisionContext": {
      "gate": "verify->release",
      "inputs": { "specId": "...", "reviewsComplete": true, "verificationPassed": true },
      "result": { "allowed": true, "blockers": [], "evidence": ["..."] },
      "reasoning": "All required reviews passing, verification passed with high confidence"
    }
  }
}
```

**Required typed events for Slice B** (these must be emitted by gate evaluator and bridge adapters):
- `session:start` — with sessionId in metadata
- `session:end` — with sessionId in metadata
- `session:gate` — with decisionContext including gate name, inputs, result
- `artifact:created` — with entityType ("intake_brief"|"solution_thesis"|"verification_result"|etc.), entityId
- `session:stage_transition` — with from, to phase names
- `session:error` — with error details

These extend the existing `EventLog.logSession()` action types (session:start, session:end, session:decision, session:error, session:gate, session:stage_transition already declared in event-log.ts).

**Relationship to Report Card:** The report card's `replaySessionId` links to this session ID. Dimensions 1-4 are scored by querying EventLog events filtered by sessionId: check for `artifact:created` events with entityType "intake_brief", "solution_thesis", then query the artifacts themselves for detail scoring.

### 6. Auto-Dogfood Entry Generation
**Trigger:** After Learning Journal update, if any pattern transitions from `recurring: false` to `recurring: true` (newly recurring).

**Format:** Appends to DOGFOOD.md matching existing format:
```markdown
## {next number}. {dimension description} (auto-detected {date})

**What happened:** {pattern detail from Learning Journal}

**Expected:** {prescription text}

**Severity:** {mapped from occurrences — see severity mapping in Section 3}

**Status:** OPEN (auto-detected by self-healing system)
```

**DOGFOOD.md creation:** If DOGFOOD.md does not exist, create it with the standard header: `# Prism Dogfood Feedback\n\nCollected during active use.\n\n---\n`

**Dedup mechanism:** Maintain `.prism/dogfood/dogfood-index.json` as machine-readable index:
```json
{
  "entries": [
    { "dimension": "research_proof", "dogfoodNumber": 12, "status": "OPEN", "createdAt": "ISO-8601" }
  ]
}
```
Before appending, check the index. If an OPEN entry exists for the same dimension, update the existing DOGFOOD.md entry's severity and occurrence count (parse by entry number) instead of creating a duplicate. DOGFOOD.md remains human-readable; the index is source of truth for dedup.

## Reviewer Concerns (unresolved after 3 review iterations)

1. **Scoring interpolation:** Dimensions define anchors at 0/5/10 but no interpolation rule. Resolution: use proportional scoring (e.g., score = 10 * high_count / total_findings for evidence_quality). Define per-dimension formulas during Slice C implementation.

2. **Manual prompt to score mapping:** A/B/C options need explicit numeric mapping (A=10, B=5, C=0). Define during Slice C.

3. **All-null edge case:** If all 6 dimensions are null (crash recovery + skipped manual), overallScore = null and HEALTH.md shows "Insufficient data."

4. **Prescription resolution streak:** Null-scored sessions (skipped manual) are ignored when counting "3 consecutive sessions >= 7" for prescription resolution.

5. **Dogfood severity escalation contradiction:** The single-fire trigger (recurring: false->true) creates entries at severity "medium" always. Simplify: dogfood entries are append-only, severity set at creation. The index tracks `occurrenceCount` for monitoring but does not trigger re-writes to DOGFOOD.md.

6. **Markdown mutation fragility:** Simplify to append-only writes for DOGFOOD.md. Use dogfood-index.json to suppress duplicate creation silently. No in-place editing of DOGFOOD.md.

7. **Checkpoint.currentSessionId:** Add `currentSessionId?: string` to existing Checkpoint type. Overwritten at session:start, set to null after report card generation.

8. **HEALTH.md is machine-generated:** Any manual edits will be overwritten. Operators wanting custom notes should use a separate file.

9. **Journal aggregation performance:** Full aggregation acceptable in v1. Add incremental counters if session count exceeds 100. Deferred optimization.

10. **HEALTH.md pass/fail threshold:** Score >= 5 = pass, < 5 = fail for the display table.

## Scope Additions (from TODO review)

### 8. Local Pattern Export
**Location:** `.prism/dogfood/export.json`
Generated on demand or after each learning journal update. Contains anonymized behavioral
patterns: dimension averages, recurring patterns, session count, trend direction. No file
paths, project names, or code content. Designed as the data shape for future upstream sharing.

### 9. Prescription Dismiss UX
Resume prompt gains a dismiss option: "D) Dismiss this prescription." Sets prescription
status to "dismissed" with timestamp. Dismissed prescriptions are excluded from future
resume prompts but retained in the file for audit.

### 10. Per-Project Report Cards
Learning Journal gains per-project filtering. `LearningJournal.update(projectId?)` can
scope to a single project. HEALTH.md gains a per-project section when multiple projects exist.
Report cards already have projectId — this is a query filter, not a new data model.

### 11. Historical Trend Visualization
**Location:** `.prism/dogfood/trends.html`
Standalone HTML file with inline SVG charts. Shows:
- Health score over time (line chart)
- Per-dimension scores over time (sparklines)
- Session count and trend arrows
Regenerated alongside HEALTH.md after each report card. Opens in any browser.
No external dependencies — pure HTML + inline SVG.
